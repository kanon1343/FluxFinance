# 開発サイクル・レビューワークフロー

## 概要

このワークフローは、タスク実装後のコードレビューを自動化し、品質基準を満たすまで修正を繰り返すプロセスを定義します。

## 実行手順

### Phase 1: タスク選択

1. `tasks.md` から未完了タスク（`[ ]`）を選択
2. 関連する要件（Requirements）を確認
3. 設計ドキュメント（design.md）を参照

### Phase 2: 実装

1. ガイドライン遵守を意識して実装
   - Server Component 優先
   - 型安全性の確保
   - クリーンコード原則
2. 実装完了後、自己チェック実施

### Phase 3: レビュー実施

レビュー実施時は以下のコマンドを使用:

```
#code-review-prompt を参照して、以下のファイルをレビューしてください:
[対象ファイルパス]
```

### Phase 4: 判定と対応

#### ✅ 承認（80-100 点）の場合

1. タスクを完了としてマーク（`[x]`）
2. 次のタスクへ進む

#### ⚠️ 条件付き承認（60-79 点）の場合

1. 「改善必須事項」を確認
2. 指摘事項を修正
3. 再度レビューを実施
4. 承認になるまで繰り返す

#### ❌ 拒否（0-59 点）の場合

1. 「改善必須事項」を詳細に確認
2. 設計レベルから見直し
3. 再実装を実施
4. 再度レビューを実施
5. 承認になるまで繰り返す

## ガイドライン逸脱の禁止事項

以下の逸脱は**絶対に許可されません**:

### TypeScript

- ❌ `any` 型の使用
- ❌ `@ts-ignore` / `@ts-nocheck` の使用
- ❌ 型定義の省略

### React 19+ / Next.js 16+

- ❌ 不要な `'use client'` の使用
- ❌ `pages/` ディレクトリの使用（App Router 必須）
- ❌ `getServerSideProps` / `getStaticProps` の使用
- ❌ クラスコンポーネントの使用

### コード品質

- ❌ 未使用のインポート/変数
- ❌ console.log のコミット（デバッグ用）
- ❌ ハードコードされた機密情報
- ❌ 重複コード（DRY 違反）

---

## レビューループの最大回数

- **条件付き承認**: 最大 3 回の修正ループ
- **拒否**: 最大 2 回の再実装ループ

上限を超えた場合は、設計の根本的な見直しが必要です。

---

## チェックポイント

タスクファイルの「チェックポイント」に到達した場合:

1. それまでの全タスクが「承認」状態であることを確認
2. 統合テストを実施
3. 問題があれば該当タスクに戻って修正

## コミットルール

### コミットメッセージ形式

```
<type>: <subject> (タスクX.Y)

<body>

<footer>
```

### Type 一覧

| Type     | 説明                 | 例                     |
| -------- | -------------------- | ---------------------- |
| feat     | 新機能追加           | feat: StockCard 実装   |
| fix      | バグ修正             | fix: API エラー修正    |
| refactor | リファクタリング     | refactor: 型定義整理   |
| test     | テスト追加・修正     | test: E2E テスト追加   |
| docs     | ドキュメント更新     | docs: README 更新      |
| style    | コードスタイル修正   | style: Biome 適用      |
| perf     | パフォーマンス改善   | perf: キャッシュ最適化 |
| build    | ビルド設定変更       | build: 依存関係更新    |
| ci       | CI/CD 設定変更       | ci: GitHub Actions     |
| chore    | その他のメンテナンス | chore: 設定ファイル    |

### Subject ルール

- 50 文字以内
- 動詞で始める（実装、追加、修正、更新）
- 句読点なし
- タスク番号を含める（例：タスク 4.2）

### Body ルール

- 変更内容の詳細を箇条書きで記載
- 「何を」「なぜ」を明確に
- レビュースコアを含める（承認の場合）

### Footer ルール

- Breaking Changes: 破壊的変更がある場合
- Closes: 関連 Issue 番号
- Reviewed-by: レビュアー情報

### コミット例

```bash
# 機能実装完了時
git commit -m "feat: StockCard コンポーネント実装完了 (タスク4.2)

- 銘柄情報表示カードの作成
- 価格変動の色分け表示（緑/赤）
- カードクリックイベントハンドリング
- Tailwind CSS によるレスポンシブデザイン

レビュースコア: 85/100点 ✅承認"

# バグ修正時
git commit -m "fix: API エラーハンドリング改善 (タスク2.1)

- Yahoo Finance API のタイムアウト処理追加
- エラーレスポンスの型安全性向上
- フォールバック値の設定

Closes #123"

# リファクタリング時
git commit -m "refactor: 型定義の整理とコード重複解消 (タスク6.1)

- YahooFinance 型定義を統一
- actions.ts の any 型を削除
- 共通ユーティリティ関数の抽出

レビュースコア: 90/100点 ✅承認"
```

### チェックポイント時のコミット

```bash
git commit -m "checkpoint: API Routes 動作確認完了 (タスク3)

- E2E テスト 11 件すべてパス
- Market API: 4つのデフォルトティッカー対応
- History API: 時間範囲別キャッシュ設定
- 型安全性とエラーハンドリング確認済み

次のフェーズ: コア UI コンポーネント実装"
```
