# 開発サイクル・レビューワークフロー

## 概要

このワークフローは、タスク実装後のコードレビューを自動化し、品質基準を満たすまで修正を繰り返すプロセスを定義します。

## 実行手順

### Phase 1: タスク選択

1. `tasks.md` から未完了タスク（`[ ]`）を選択
2. 関連する要件（Requirements）を確認
3. 設計ドキュメント（design.md）を参照

### Phase 2: 実装

1. ガイドライン遵守を意識して実装
   - Server Component 優先
   - 型安全性の確保
   - クリーンコード原則
2. 実装完了後、自己チェック実施

### Phase 3: レビュー実施

レビュー実施時は以下のコマンドを使用:

```
#code-review-prompt を参照して、以下のファイルをレビューしてください:
[対象ファイルパス]
```

### Phase 4: 判定と対応

#### ✅ 承認（80-100 点）の場合

1. タスクを完了としてマーク（`[x]`）
2. 次のタスクへ進む

#### ⚠️ 条件付き承認（60-79 点）の場合

1. 「改善必須事項」を確認
2. 指摘事項を修正
3. 再度レビューを実施
4. 承認になるまで繰り返す

#### ❌ 拒否（0-59 点）の場合

1. 「改善必須事項」を詳細に確認
2. 設計レベルから見直し
3. 再実装を実施
4. 再度レビューを実施
5. 承認になるまで繰り返す

## ガイドライン逸脱の禁止事項

以下の逸脱は**絶対に許可されません**:

### TypeScript

- ❌ `any` 型の使用
- ❌ `@ts-ignore` / `@ts-nocheck` の使用
- ❌ 型定義の省略

### React 19+ / Next.js 16+

- ❌ 不要な `'use client'` の使用
- ❌ `pages/` ディレクトリの使用（App Router 必須）
- ❌ `getServerSideProps` / `getStaticProps` の使用
- ❌ クラスコンポーネントの使用

### コード品質

- ❌ 未使用のインポート/変数
- ❌ console.log のコミット（デバッグ用）
- ❌ ハードコードされた機密情報
- ❌ 重複コード（DRY 違反）

---

## レビューループの最大回数

- **条件付き承認**: 最大 3 回の修正ループ
- **拒否**: 最大 2 回の再実装ループ

上限を超えた場合は、設計の根本的な見直しが必要です。

---

## チェックポイント

タスクファイルの「チェックポイント」に到達した場合:

1. それまでの全タスクが「承認」状態であることを確認
2. 統合テストを実施
3. 問題があれば該当タスクに戻って修正

## 使用例

### 例 1: StockCard コンポーネント実装後

```
タスク 4.2 StockCard コンポーネントの実装が完了しました。

#code-review-prompt を参照して、以下のファイルをレビューしてください:
- src/components/StockCard.tsx
```

### 例 2: 条件付き承認後の修正

```
前回のレビューで条件付き承認（72点）でした。
以下の指摘事項を修正しました:
- any型をStockDataType型に変更
- コンポーネントを分割してSRP遵守

再度レビューをお願いします:
- src/components/StockCard.tsx
- src/components/PriceDisplay.tsx
```
