# コードレビュープロンプト 2026

## 概要

このプロンプトは、TypeScript、React 19+、Next.js 16+ のベストプラクティスに基づいたコードレビューを実施するためのガイドラインです。ソフトウェア業界の著名人とその代表的書籍に基づくスコアリングシステムを採用し、コード品質を客観的に評価します。

## スコアリングシステム

### 評価カテゴリと著名人・書籍

各カテゴリは 0-10 点で評価し、合計 100 点満点です。

| カテゴリ           | 著名人           | 代表書籍                   | 配点  |
| ------------------ | ---------------- | -------------------------- | ----- |
| テスト駆動開発     | Kent Beck        | 『テスト駆動開発』         | 10 点 |
| クリーンコード     | Robert C. Martin | 『Clean Code』             | 15 点 |
| リファクタリング   | Martin Fowler    | 『リファクタリング』       | 10 点 |
| ドメイン駆動設計   | Eric Evans       | 『ドメイン駆動設計』       | 10 点 |
| 型安全性           | Anders Hejlsberg | TypeScript 設計思想        | 15 点 |
| コンポーネント設計 | Dan Abramov      | React 設計原則             | 15 点 |
| パフォーマンス     | Addy Osmani      | 『JavaScript Performance』 | 10 点 |
| セキュリティ       | OWASP            | セキュリティガイドライン   | 10 点 |
| アクセシビリティ   | W3C WAI          | WCAG 2.2                   | 5 点  |

### 判定基準

| 判定            | スコア範囲 | 説明                           |
| --------------- | ---------- | ------------------------------ |
| ✅ 承認         | 80-100 点  | マージ可能。軽微な改善提案のみ |
| ⚠️ 条件付き承認 | 60-79 点   | 指摘事項の修正後にマージ可能   |
| ❌ 拒否         | 0-59 点    | 重大な問題あり。再実装が必要   |

## 評価チェックリスト

### 1. Kent Beck スコア（テスト駆動開発）- 10 点

```
□ テストが先に書かれている、または十分なテストカバレッジがある
□ Red-Green-Refactor サイクルが守られている
□ テストは独立して実行可能
□ テスト名が振る舞いを明確に表現している
□ エッジケースがカバーされている
```

**評価基準:**

- 10 点: テストファースト、カバレッジ 80%以上
- 7 点: テストあり、主要パスをカバー
- 4 点: 最低限のテストのみ
- 0 点: テストなし

### 2. Robert C. Martin スコア（クリーンコード）- 15 点

```
□ 関数は単一責任を持つ（20行以内推奨）
□ 変数名・関数名が意図を明確に表現
□ コメントは「なぜ」を説明（「何を」はコードで表現）
□ ネストは3レベル以内
□ 重複コードがない（DRY原則）
□ 早期リターンパターンを使用
```

**評価基準:**

- 15 点: すべての原則を遵守
- 10 点: 軽微な違反のみ
- 5 点: 複数の違反あり
- 0 点: 可読性が著しく低い

### 3. Martin Fowler スコア（リファクタリング）- 10 点

```
□ コードの臭い（Code Smell）がない
□ 適切な抽象化レベル
□ 条件分岐がポリモーフィズムで置換可能な場合は置換
□ マジックナンバー/文字列が定数化されている
□ 長いパラメータリストがオブジェクトに置換されている
```

**評価基準:**

- 10 点: リファクタリング不要
- 7 点: 軽微なリファクタリング推奨
- 4 点: 中程度のリファクタリング必要
- 0 点: 大規模なリファクタリング必要

### 4. Eric Evans スコア（ドメイン駆動設計）- 10 点

```
□ ドメインモデルが明確に定義されている
□ ユビキタス言語が一貫して使用されている
□ 境界づけられたコンテキストが適切
□ 値オブジェクトとエンティティの区別が明確
□ リポジトリパターンが適切に使用されている
```

**評価基準:**

- 10 点: DDD 原則を完全に遵守
- 7 点: 主要な概念を適用
- 4 点: 部分的な適用
- 0 点: ドメインモデルが不明確

### 5. Anders Hejlsberg スコア（型安全性）- 15 点

```
□ any型を使用していない
□ 適切な型定義（interface/type）
□ ジェネリクスの適切な使用
□ null/undefinedの適切なハンドリング
□ 型ガードの使用
□ Zodによるランタイムバリデーション
```

**評価基準:**

- 15 点: 完全な型安全性
- 10 点: 軽微な型の問題のみ
- 5 点: any 型の使用あり
- 0 点: 型安全性が欠如

### 6. Dan Abramov スコア（コンポーネント設計）- 15 点

```
□ Server Component優先（デフォルト）
□ Client Componentは必要な場合のみ（'use client'）
□ コンポーネントは単一責任
□ Props drilling を避けている
□ 適切なコンポーネント分割
□ カスタムフックによるロジック分離
```

**評価基準:**

- 15 点: React 19+のベストプラクティス完全遵守
- 10 点: 軽微な設計上の問題
- 5 点: Client Component 過剰使用
- 0 点: コンポーネント設計が不適切

### 7. Addy Osmani スコア（パフォーマンス）- 10 点

```
□ 不要な再レンダリングを防止（memo, useMemo, useCallback）
□ 動的インポートによるコード分割
□ 画像最適化（next/image使用）
□ バンドルサイズの最適化
□ Core Web Vitals を考慮
```

**評価基準:**

- 10 点: パフォーマンス最適化完了
- 7 点: 主要な最適化実施済み
- 4 点: 基本的な最適化のみ
- 0 点: パフォーマンス問題あり

### 8. OWASP スコア（セキュリティ）- 10 点

```
□ 入力バリデーション実施
□ XSS対策（dangerouslySetInnerHTML不使用）
□ CSRF対策
□ 環境変数の適切な管理
□ 認証・認可の適切な実装
```

**評価基準:**

- 10 点: セキュリティベストプラクティス遵守
- 7 点: 軽微なセキュリティ懸念
- 4 点: 中程度のセキュリティリスク
- 0 点: 重大なセキュリティ脆弱性

### 9. W3C WAI スコア（アクセシビリティ）- 5 点

```
□ セマンティックHTML使用
□ ARIA属性の適切な使用
□ キーボードナビゲーション対応
□ 色のコントラスト比確保
□ スクリーンリーダー対応
```

**評価基準:**

- 5 点: WCAG 2.2 AA 準拠
- 3 点: 基本的なアクセシビリティ対応
- 1 点: 最低限の対応
- 0 点: アクセシビリティ未対応

## Next.js 16+ / React 19+ 固有チェック

### App Router 必須チェック

```
□ app/ ディレクトリ構造を使用
□ layout.tsx でレイアウト定義
□ loading.tsx でローディングUI
□ error.tsx でエラーハンドリング
□ Server Actions の適切な使用
```

### データフェッチング

```
□ Server Component でのデータフェッチング優先
□ fetch() に適切なキャッシュ設定
□ revalidate オプションの適切な設定
□ TanStack Query はClient Componentのみ
```

### スタイリング（Tailwind CSS 4.0+）

```
□ cn() ユーティリティでクラス結合
□ レスポンシブデザイン（sm:, md:, lg:）
□ ダークモード対応（dark:）
□ カスタムクラスは最小限
```

## 使用方法

1. タスク実装完了後、このプロンプトを参照してレビューを実施
2. 各カテゴリのチェックリストを確認
3. スコアを算出し、判定を決定
4. 「条件付き承認」または「拒否」の場合は修正を実施
5. 「承認」になるまでステップ 1-4 を繰り返す
