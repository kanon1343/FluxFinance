# Issue作成ガイドライン

このスキルは、プロジェクトにおいて新しい機能要求やタスクを定義する際（Issueを作成する際）の標準的なフォーマットと手順を定めています。世の中のAgile/GitHubベストプラクティスに基づいています。

## 必須要件
1. **言語:** 必ず **日本語** で記述すること。
2. **必須項目:** 「概要」と「受け入れ条件」は必ず含めること。
3. **ユーザー中心:** 可能であればユーザーストーリーを含め、誰のどのような課題を解決するかを明確にすること。

## 標準テンプレートと作成手順
Issueを作成する際は、Markdownファイルとしてローカルに保存する**のではなく**、必ずGitHub CLI (`gh issue create`) などのコマンドを使用して実際のリポジトリにIssueを作成してください。

```bash
# 作成コマンドの例
gh issue create --title "[Feature] 機能のタイトル" --body-file path/to/body.md
# または
gh issue create --title "[Feature] 機能のタイトル" --body "..."
```

作成するIssueのBodyには、以下のMarkdownテンプレートを適用してください。

```markdown
## 概要
（このIssueで実現したい機能や変更内容の全体像を簡潔に記述します）


## 背景・解決したい課題
（なぜこの機能が必要なのか、ユーザーが直面している問題は何かを記述します。現状の何が不便かを明確にします）

## ユーザーストーリー
* **As a (〜として)**: [例: 個人投資家]
* **I want to (〜したい)**: [例: 最新ニュースの感情分析を見たい]
* **So that (なぜなら)**: [例: 株価変動の理由を瞬時に理解したいから]

## 技術的アプローチ・詳細
（APIの仕様、アーキテクチャの変更点、依存関係など、開発者向けの技術的なアプローチを記述します）

## 受け入れ条件 (Acceptance Criteria)
（このIssueが完了したとみなすための具体的な条件をチェックリスト形式で記述します）
- [ ] {条件1: 例 - APIルートが作成され、200 OKを返すこと}
- [ ] {条件2: 例 - 画面上にバッジが表示されること}
```
